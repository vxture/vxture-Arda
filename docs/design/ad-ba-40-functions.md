## 0. 两层区分：看和组织 vs 功能

| 层 | 是什么 | 归谁 / 何时 |
|---|---|---|
| **看和组织** | 功能怎么切换、页面怎么归组、导航 | header **九宫 launcher**，后续逐步落地（`ba-11`/`ba-2x`） |
| **功能**（本系列 `ba-4x`） | 每个功能**端到端能不能用** | **现在设计**，与"在哪显示"无关 |

**为什么按功能设计、不按导航**：导航解决"看得到"，功能解决"用得了"。一个功能哪怕导航做得再漂亮，若"目标定了没执行器""结果出了服务用不上""改了没审计"——就是**断链 = 没贯通 = 没真正实现**。本系列**逐功能沿生命周期查断链、给实现**。

## 2. 功能设计模板（每个 `ba-4x` 统一结构）

1. **功能定义**：一句话 + 它保证什么。
2. **贯通链（目标 → 过程 → 结果 → 服务 → 监管）**：逐环写「做什么 · 靠什么实现（实体/操作）· 怎么接下一环」。
3. **断链清单**：当前哪些环没接通 / 没实现（编号 `<X>-BL<n>`，喂 §3 与 `ba-30`）。
4. **数据模型（实现）**：实体 + 字段 delta（以 da 维度为准）+ 关键操作。
5. **依赖**：其他功能 / 平台服务。
6. **门控**：能力键（`arda.*`，**不涉及导航**）。

> **贯通判据**：五环每一环都有实现、且**上一环产出正好是下一环输入**（目标产规则→过程消费规则产结果→结果供服务与监管）。任一环缺实现或接口错位 = 断链。

## 3. 断链总看板（跨功能 · 实施直接输入）

汇总各功能的断链，按优先级排；这张表**就是 `ba-30` 实施计划的 backlog**。（随各功能 `ba-4x` 补齐；当前仅 `ba-45` 数据质量已填。）

| 编号 | 功能 | 断链（哪一环没接通） | 优先级 | 依赖 |
|---|---|---|---|---|
| `Q-BL1` | 质量 | 无真实调度：`QualityResult` 靠 seed，质检跑不起来（过程环） | 高 | scheduling（`ba-41`/future） |
| `Q-BL2` | 质量 | 服务准入未接：`DataService` 不读质量分（服务环） | 高 | `ba-48` |
| `Q-BL3` | 质量 | 审计未接：规则变更 / 告警不落 `AuditLog`（监管环） | 中 | `ba-49`/admin |
| `Q-BL4` | 质量 | 阈值 / SLA 未建模：`QualityRule` 无 threshold（目标环，可用 config 承载） | 中 | da（可选迁移） |
| ... | | 其余功能的断链随 `ba-42..49` 补入 | | |

> 看板列义：**断链 = 功能没贯通的点**；实施就是**把断链一条条接通**。`ba-30` 按此表分阶段排期。
